# 后续开发功能清单

本文记录当前需要继续完善的上游管理相关功能，按优先级拆成可实现、可验收的任务。

## P0 上游接口访问优化

### 1. 上游 token 缓存

问题：

- 当前拉取上游分组、上游密钥等数据时，不应每次都重新登录获取 token。
- 只有上游返回 token 失效、未授权或明确认证失败时，才重新登录并刷新 token。

实现要点：

- 在上游 provider 适配层增加按 provider slug 缓存的 token 信息。
- token 缓存至少包含 access token、过期时间或最后刷新时间。
- 拉取 key、group、model-square 时优先使用缓存 token。
- 遇到 401、403 或上游约定的 token 失效错误时，清理缓存并重试一次登录。
- 日志和接口响应不得泄露 token、cookie、密码或完整 key。

验收标准：

- 同一个 provider 连续拉取数据时，只触发一次登录。
- token 失效后会自动重新登录并重试原请求。
- 并发请求不会同时触发大量重复登录。

## P1 上游分组页面优化

### 2. 复刻旧版上游分组展示

问题：

- 当前页面的“本地分组”列使用下拉选择，操作意图不够清晰。
- “状态”列含义偏技术化，不能直接表达用户关心的结果。
- 参考项目 `D:\codingspace\workspace-zhongyj\sub2api` 中的上游分组页面更接近目标形态。

目标：

- 保留现有后端对默认上游分组的对账能力。
- 前端展示改为更直观的只读匹配结果，未匹配行只提供“同步”操作。
- 状态文案改成明确业务状态，例如：未匹配、已匹配、倍率风险、倍率正常。

实现要点：

- 上游分组页面展示：上游分组名称、上游倍率、上游密钥数量、本地分组名称、本地倍率、业务状态、操作。
- 未匹配时显示“同步”按钮，创建本地分组并写入映射。
- 已匹配时不再把本地分组作为默认可编辑下拉；如仍需手动映射，后续单独做“映射管理”入口。
- 搜索支持上游分组、本地分组和 provider 信息。

验收标准：

- 普通查看场景不需要通过下拉理解匹配关系。
- 未匹配上游分组可以同步为本地分组。
- 倍率风险能明确提示上游倍率与本地倍率差异。

## P1 上游账号页面优化

### 3. 展示已匹配账号绑定的本地分组和倍率

问题：

- 当前上游账号页面显示单个“本地分组”，容易误解为账号只有一个绑定分组。
- 当上游密钥名称与本地账号名称匹配成功后，用户真正关心的是该本地账号当前绑定了哪些分组，以及这些分组倍率是否低于上游倍率。
- “动作”列目前含义不够直观，需要更明确地展示同步将做什么。

匹配规则：

- 本地账号名 = 上游 provider 的 `account_name_prefix` + 上游密钥名称。
- 匹配时去空格，不区分大小写。
- 如果同一个规范化名称匹配多个本地账号，标记冲突，不自动处理。

实现要点：

- 后端预览结果为已匹配账号返回当前绑定的全部本地分组：分组 ID、名称、倍率、是否低于上游倍率。
- 前端“本地分组”列改为“账号绑定分组”，以标签形式展示全部分组和倍率。
- 存在低倍率分组时，在同一行明确展示“将解绑”分组。
- “动作”列文案保持：创建、更新、无需处理、跳过、冲突；低倍率时额外显示风险说明。

验收标准：

- 已匹配账号能看到全部绑定分组和倍率。
- 低于上游倍率的分组能被标出，并与同步后解绑行为一致。
- 创建账号场景仍显示将绑定的目标分组。

## P2 上游账号倍率守护

### 4. 定时解绑低倍率分组

问题：

- 当前账号倍率保护主要依赖手动执行上游账号同步。
- 需要一个定时守护任务自动检查：如果上游密钥倍率高于本地账号已绑定分组倍率，则解绑该账号和低倍率分组。

规则：

- 只处理已经按名称匹配成功的本地账号。
- 上游倍率大于本地分组倍率时，解绑该账号与该低倍率分组。
- 高于或等于上游倍率的本地分组保留。
- 任务间隔可配置，默认关闭，避免部署后自动修改绑定关系。
- 需要记录运行状态和解绑审计。

实现要点：

- 复用现有 `UpstreamAccountSyncService` 中低倍率分组识别和解绑逻辑，避免另写一套规则。
- 新增账号倍率守护配置：enabled、interval_seconds、last_run_at、last_run_status、last_run_message。
- 新增定时 scheduler，进程内防重入。
- 管理页展示最近运行状态、解绑数量和最近审计记录。

验收标准：

- 配置关闭时不会执行解绑。
- 配置开启且到达间隔后会自动检查并解绑低倍率分组。
- 失败时写入最近失败状态，不影响下一轮执行。
- 守护逻辑与手动同步的解绑结果一致。

## 推荐实施顺序

1. 补齐上游账号预览的“账号绑定分组”字段和页面展示。
2. 调整上游分组页面，去掉默认下拉交互，改成只读匹配结果加同步操作。
3. 增加上游 token 缓存，降低上游登录频率。
4. 抽取账号倍率守护的可复用执行入口，再接入定时配置和 scheduler。
