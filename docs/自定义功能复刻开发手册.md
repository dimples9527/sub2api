# 自定义功能复刻开发手册

本文记录当前分支相对主线新增或迁移的自定义能力。目标不是产品介绍，而是给以后重新开发、迁移到新分支、或从零复刻时使用。

## 使用方式

1. 先阅读“全局接线点”，理解哪些现有文件必须少量修改。
2. 按“推荐复刻顺序”逐个模块实现。
3. 每实现一个模块，跑对应测试和构建。
4. 不要直接把所有逻辑塞进现有大文件；优先新建模块，只在路由、菜单、依赖注入、API barrel、i18n 做接线。

## 推荐复刻顺序

1. 前端基础字号和帮助中心。
2. 上游配置管理。
3. 上游分组匹配和倍率修正。
4. 上游账号同步。
5. 模型监控页面。
6. 模型广场。
7. 充值页面样式和充值金额配置。
8. Docker 构建和 CentOS 部署文档。

## 2. 帮助中心

目的：迁移参考文档入口，给普通用户提供内置帮助页面。

主要文件：

- `frontend/public/help.html`
- `frontend/src/views/user/HelpView.vue`
- `frontend/src/router/index.ts`
- `frontend/src/components/layout/AppSidebar.vue`
- `frontend/src/i18n/locales/zh.ts`
- `frontend/src/i18n/locales/en.ts`
- `backend/internal/config/config.go`
- `backend/internal/server/middleware/security_headers_test.go`
- `deploy/config.example.yaml`

路由：

- `/help`

实现要点：

- `HelpView.vue` 作为应用内页面入口。
- `help.html` 放在 `public`，用于承载静态参考文档内容。
- 菜单只增加入口，不改已有用户菜单结构。
- 帮助页通过同源 iframe 加载 `/help.html`，默认 CSP 的 `frame-src` 必须包含 `'self'`。
- 如果部署环境使用了显式 `security.csp.policy`，也要同步把 `frame-src` 改为 `frame-src 'self' ...`，只改代码默认值不会覆盖现有配置文件。
- `deploy/config.example.yaml` 需要和默认 CSP 保持一致，避免新部署继续复制旧策略。
- `/help.html` 自身不能继续下发 `frame-ancestors 'none'` 或 `X-Frame-Options: DENY`，否则子页面会拒绝被 iframe；中间件需要对 `/help.html` 使用 `frame-ancestors 'self'` 和 `X-Frame-Options: SAMEORIGIN`。
- 如果浏览器报错里的 Framing URL 是站点根路径（例如 `https://api.sunshinelink.online/`），优先检查线上是否还存在旧的自定义菜单 URL 或旧前端包；内置帮助菜单应进入 `/help`，iframe 子页面应是 `/help.html`。

验证：

- 路由可访问。
- 普通用户侧菜单能看到帮助入口。
- `pnpm.cmd run build`
- `cd backend && go test ./internal/server/middleware`

目的：新增独立“上游管理”模块，支持维护 Sub2API 和 NewAPI 两种上游配置，后续可扩展更多类型。

后端主要文件：

前端主要文件：

- `frontend/src/api/admin/upstreamProviders.ts`
- `frontend/src/views/admin/upstream-management/UpstreamProvidersView.vue`
- `frontend/src/router/index.ts`
- `frontend/src/components/layout/AppSidebar.vue`

配置存储：

- setting key: `upstream_provider_configs`

菜单和页面：

- 管理员菜单：`上游管理`
- 子菜单：`上游配置`
- 前端路由：`/admin/upstream-management/providers`

后端接口：

- `GET /admin/upstream-management/providers`
- `POST /admin/upstream-management/providers`
- `POST /admin/upstream-management/providers/test`
- `PUT /admin/upstream-management/providers/:slug`
- `DELETE /admin/upstream-management/providers/:slug`
- `POST /admin/upstream-management/providers/:slug/default`
- `POST /admin/upstream-management/providers/:slug/test`
- `GET /admin/upstream-management/providers/:slug/keys`

Provider 字段：

- `type`: `sub2api`、`newapi`，后续允许扩展字符串。
- `slug`: 稳定标识，更新时不可改。
- `name`
- `enabled`
- `is_default`
- `base_url`
- `login_url`
- `api_keys_url`
- `groups_url`
- `email`
- `username`
- `password`
- `password_configured`
- `account_name_prefix`
- `temp_disable_minutes`

关键规则：

- 密码不回显，更新时密码留空表示保留原密码。
- 同一时间只有一个默认上游。
- 本地分组和后续模型广场只基于默认上游；上游账号同步只基于非默认上游。
- 测试按钮必须先执行登录，再获取密钥；NewAPI 还需要拉取分组接口并合并倍率。
- `sub2api` 获取密钥只需要一个 key 接口。
- `newapi` 获取密钥需要 key 接口和 groups 接口两个数据源。
- `sub2api` 登录响应要兼容 `data.access_token`、`data.token`、顶层 `access_token`、顶层 `token`；2.0 登录接口通常返回 `data.access_token`。
- `sub2api` 2.0 的默认登录接口建议提示为 `/api/v1/auth/login`；`/api/user/login` 更偏 NewAPI 风格。
- 测试结果按 `login`、`keys`、`groups` 分阶段展示，并返回解析后的样例 key。
- 日志和响应不得泄露密码、token、cookie、完整 key。

适配器扩展方式：

1. 新增 provider type 常量。
2. 新增 `UpstreamProviderAdapter` 实现。
3. 在 `UpstreamProviderAdapterRegistry` 注册。
4. 前端表单增加该类型对应字段说明。

核心测试：

- `backend/internal/service/upstream_provider_test.go`
- `backend/internal/handler/admin/upstream_provider_handler_test.go`

## 4. 上游分组

目的：新增管理员菜单“上游分组”，展示默认上游分组和本地分组匹配结果，对比倍率，并支持把本地低倍率修正到上游倍率。
未匹配的上游分组可以通过页面上的“同步”按钮创建为本地分组，并自动保存映射关系。

后端主要文件：

- `backend/internal/service/upstream_management.go`
- `backend/internal/service/upstream_group_rate_fix_scheduler.go`
- `backend/internal/handler/admin/upstream_management_handler.go`
- `backend/internal/server/routes/upstream_management.go`

前端主要文件：

- `frontend/src/api/admin/upstreamManagement.ts`
- `frontend/src/views/admin/upstream-management/UpstreamGroupsView.vue`

存储：

- `upstream_group_mappings`
- `upstream_group_rate_fix_records`
- `upstream_group_rate_fix_config`

`upstream_group_rate_fix_config` 示例：

```json
{
  "enabled": false,
  "interval_seconds": 3600,
  "last_run_at": "2026-06-11T12:00:00Z",
  "last_run_status": "success",
  "last_run_message": "",
  "updated_at": "2026-06-11T12:00:00Z"
}
```

默认值：

- `enabled=false`
- `interval_seconds=3600`
- `last_run_*` 为空

页面：

- `/admin/upstream-management/groups`

接口：

- `GET /admin/upstream-management/groups`
- `PUT /admin/upstream-management/groups/mappings`
- `POST /admin/upstream-management/groups/local-groups`
- `GET /admin/upstream-management/groups/rate-fix-config`
- `PUT /admin/upstream-management/groups/rate-fix-config`
- `POST /admin/upstream-management/groups/rate-fixes`

匹配规则：

- 只使用默认上游。
- 上游分组和本地分组的对应不使用 `account_name_prefix`。
- 手动映射优先。
- 未手动映射时按规范化名称匹配。
- 规范化规则：去首尾空白、合并连续空白、转小写。

同步创建规则：

- 只允许从默认上游分组创建本地分组。
- 后端必须重新读取默认上游密钥列表，校验传入分组确实存在于默认上游。
- 页面只在未匹配行显示“同步”按钮。
- 同步弹窗默认使用上游倍率，允许确认前调整本地倍率。
- 后端创建 `platform=openai`、`status=active`、`subscription_type=standard` 的本地分组。
- 创建成功后写入 `upstream_group_mappings`，把该上游分组映射到新建本地分组。
- 如果本地已存在同名分组，或规范化名称已经能匹配到本地分组，返回 `GROUP_EXISTS`，不重复创建。

页面筛选：

- 支持按上游分组、本地分组、来源关键词搜索。
- 支持按匹配状态筛选：全部、已匹配、未匹配。
- 支持按倍率状态筛选：全部、倍率风险、倍率正常。

倍率规则：

- 展示上游倍率和本地倍率。
- 如果上游倍率大于本地倍率，标记风险。
- 执行修正时，把本地分组倍率提高到上游倍率。
- 修正需要记录修改历史，包括本地分组、上游分组、旧倍率、新倍率、时间。

自动修正规则：

- 自动修正默认关闭，避免部署后自动修改本地分组倍率。
- 管理后台可配置是否启用和执行间隔，间隔单位为秒，最小值为 1。
- 秒级间隔适用于小规模或临时高频检查；生产环境应结合上游 key 数量、上游接口限流和数据库负载谨慎设置。
- 后台任务只调用现有 `ApplyRateFixes()`，不重写倍率修复逻辑。
- 后台任务只处理默认上游、已匹配分组、上游倍率大于本地倍率的场景。
- 最近执行时间、最近状态和最近错误信息写入 `upstream_group_rate_fix_config`。
- 同一进程内使用运行锁防止重叠执行；多副本部署如需全局互斥，后续再增加 Redis/DB 分布式锁。

调度实现：

- `UpstreamGroupRateFixScheduler` 启动后每秒检查一次配置。
- 配置关闭时只读取配置，不执行修复。
- 配置开启后，根据 `interval_seconds` 和进程内 `lastRunAt` 判断是否到期。
- 单次执行有 2 分钟超时，避免上游接口卡住后长期占用任务。
- `Start()` 必须幂等，防止 Wire 或测试重复启动时注册多个后台 goroutine。
- `Stop()` 必须纳入 server cleanup，保证退出时后台任务停止。

生效与缓存边界：

- 修复成功后会更新本地 `groups.rate_multiplier`。
- 更新后会调用 `InvalidateAuthCacheByGroupID`，清理该分组下 API Key 的认证缓存；认证缓存里包含分组倍率快照。
- 已经进入处理流程的请求仍可能使用旧倍率，这是现有请求生命周期决定的。
- `userGroupRateResolver` 还有 30 秒级别的最终倍率缓存。如果之前缓存过某个用户/分组的倍率，新倍率可能有短暂延迟。
- 第一版不改 `userGroupRateResolver`，也不新增跨模块缓存清理接口。这样保持改动范围小，行为与后台手动修改分组倍率保持一致。

核心测试：

- `backend/internal/service/upstream_management_test.go`
- `backend/internal/service/upstream_group_rate_fix_scheduler_test.go`
- `backend/internal/handler/admin/upstream_management_handler_test.go`

建议补充的测试点：

- 配置默认关闭，默认间隔为 3600 秒。
- 保存配置时校验 `interval_seconds >= 1`。
- 自动任务关闭时不执行修复。
- 自动任务未到间隔时不重复执行。
- 前一次自动任务未结束时，下一次到期检查应跳过，不能并发执行。
- 自动任务失败时写入 `last_run_status=failed` 和错误信息。
- 修复倍率后调用 `InvalidateAuthCacheByGroupID`。

## 5. 上游账号同步

目的：基于非默认上游的 key 列表预览并同步本地账号，处理分组绑定和倍率保护。默认上游只用于“上游分组”页面和模型广场，不作为“上游账号”页面的数据源。

后端主要文件：

- `backend/internal/service/upstream_account_sync.go`
- `backend/internal/handler/admin/upstream_account_sync_handler.go`
- `backend/internal/server/routes/upstream_management.go`

前端主要文件：

- `frontend/src/api/admin/upstreamAccountSync.ts`
- `frontend/src/views/admin/upstream-management/UpstreamAccountsView.vue`

存储：

- `upstream_account_sync_records`

页面：

- `/admin/upstream-management/accounts`

接口：

- `GET /admin/upstream-management/accounts/sync-preview`
- `POST /admin/upstream-management/accounts/sync`
- `GET /admin/upstream-management/accounts/sync-records`

请求参数：

```json
{
  "create_missing": true,
  "update_existing": true,
  "apply_rate_guard": true
}
```

账号匹配规则：

- 只读取已启用的非默认上游配置；默认上游不展示、不同步。
- 上游适配器返回的 `upstream_key_name` 必须保留上游原始 key 名，不提前加本地账号前缀。
- 本地账号名 = `account_name_prefix + upstream_key_name`
- 先精确匹配账号名，再按规范化名称匹配。
- 一个规范化名称匹配多个本地账号时，标记 `conflict`，不自动处理。

创建账号规则：

- 只在上游 key 已匹配到本地分组时创建。
- 第一版创建 `platform=openai`、`type=apikey` 的账号。
- `credentials.api_key` 使用上游 key。
- `credentials.base_url` 使用该非默认上游自己的 `base_url`。
- `group_ids` 绑定匹配到的本地分组。
- `extra.upstream_*` 记录上游来源、上游分组、上游倍率和同步时间。

更新账号规则：

- 只更新托管字段：API key、base_url、group_ids、`extra.upstream_*`。
- 不覆盖用户手工维护的并发、优先级、代理、模型映射、限额、临时禁用等字段。
- 非 OpenAI API Key 账号不写入上游 key。

倍率保护规则：

- 对已匹配账号，检查账号绑定的本地分组倍率。
- 本地分组倍率低于上游倍率时，执行同步可解绑低倍率分组。
- 满足上游倍率要求的高倍率分组保留。

核心测试：

- `backend/internal/service/upstream_account_sync_test.go`
- `backend/internal/handler/admin/upstream_account_sync_handler_test.go`

## 8. 充值页面样式和充值金额配置

目的：优化用户充值界面，并把充值金额预设改成后台可配置。

- `backend/internal/service/payment_config_service.go`


## 9. Docker 构建和部署文档

目的：支持源码构建部署时，前端能解析根目录 `docs/legal` 下的合规文档，并提供 CentOS 无域名/域名部署手册。

主要文件：

- `Dockerfile`
- `deploy/Dockerfile`
- `.dockerignore`
- `deploy/DEPLOY_CENTOS_NO_DOMAIN_CN.md`
- `deploy/DEPLOY_CENTOS_DOMAIN_CADDY_CN.md`

构建规则：

- 前端构建阶段必须包含：

```dockerfile
COPY frontend/ ./
COPY docs/legal/ /app/docs/legal/
RUN pnpm run build
```

- `.dockerignore` 继续忽略大部分 docs，但必须放行：

```dockerignore
docs/*
!docs/legal/
docs/legal/*
!docs/legal/admin-compliance.*.md
```

- `deploy/Dockerfile` 固定 `pnpm@9`，不要使用 `pnpm@latest`，避免 frozen lockfile 因 pnpm 新版本配置规则变化失败。

CentOS 无域名部署文档要点：

- 本地打包源码，不上传 `.git`、`node_modules`、构建产物。
- 服务器目录：

```text
/opt/sub2api/repo
/opt/sub2api/runtime
```

- runtime 中创建 `docker-compose.build.yml`，使用：

```yaml
build:
  context: ../repo
  dockerfile: Dockerfile
```

- 如果日志仍显示 `frontend-builder 7/7` 且 `RUN pnpm run build` 是第 32 行，说明服务器 `/opt/sub2api/repo/Dockerfile` 仍是旧版本，需要重新打包上传。

验证：

- `docker build -t sub2api-custom:local .`
- `docker build -f deploy/Dockerfile -t sub2api-custom:deploy-local .`

## 10. 合规文档构建注意点

当前前端有组件通过 raw import 引入根目录文档：

- `docs/legal/admin-compliance.zh.md`
- `docs/legal/admin-compliance.en.md`

引用方包括：

- `frontend/src/components/admin/AdminComplianceDialog.vue`
- `frontend/src/views/public/LegalDocumentView.vue`

因此任何 Docker 构建上下文都必须包含 `docs/legal`。只复制 `frontend/` 会导致 Vite/Rollup 报：

```text
Could not resolve "../../../../docs/legal/admin-compliance.zh.md?raw"
```

## 12. 已知取舍

- 上游配置、分组、账号同步都优先复用 setting 存储，避免新增数据库表。
- 默认上游是分组匹配和模型广场的数据源；上游账号同步只读取非默认上游。
- 上游分组自动修正第一版只做进程内防重入，不做分布式锁；倍率更新后复用现有认证缓存失效路径，允许极短时间的旧倍率缓存窗口。
- 模型广场第一版只挂管理员菜单，普通用户入口暂不开放。
- 上游账号同步第一版只创建 OpenAI API Key 账号。
- Docker 构建只放行合规文档，不把整个 `docs/` 打进构建上下文。
- 部署文档目前以 CentOS + Docker Compose + SCP 源码包为主。
- 帮助中心 iframe 依赖 CSP `frame-src 'self'`。如果线上 `config.yaml` 仍保留旧的显式 CSP，需要手动同步配置后重启服务。
