Detailed Design · 2026-06-10

上游管理、模型广场、LLM 监控与充值配置详细设计

本文档用于重新开发相关功能，范围限定为上游管理/模型广场、LLM 监控、充值金额可配置、账号倍率守护、帮助文档。 设计目标是把现有功能重新拆成边界清晰、配置可控、可测试、可迭代的模块。

1. 背景与目标

当前分支已经实现过一轮相关功能，但存在几个重新开发时需要规避的问题：后端模型广场 handler 承载了过多职责，前端管理页面较重，配置项存在新旧命名兼容，部分功能和展示逻辑耦合较深。 新设计不直接复刻文件结构，而是保留业务能力，重新划分模块边界。

核心目标

  • 管理员可以配置多个上游管理源。
  • 系统可以拉取上游模型、分组、API Key 摘要。
  • 用户可以查看模型广场中的可用模型和价格信息。
  • 管理员可以查看上游分组和本地分组匹配关系。
  • LLM 监控数据通过后端代理、过滤、脱敏后展示。
  • 充值金额档位和首充优惠可配置。
  • 账号倍率守护可以发现并处理成本倒挂风险。
  • 帮助文档作为独立静态文档在用户侧展示。

非目标

  • 第一版不做帮助文档后台 CMS 编辑。
  • 第一版不做复杂上游账号和本地账号的手动映射系统。
  • 第一版不把 LLM 监控大屏重写成完整 Vue 应用。
  • 第一版不改变现有支付订单核心流程，只扩展档位和到账金额计算。

2. 总体架构

后端按配置、客户端、业务服务、HTTP handler 分层。handler 只负责鉴权后的请求解析和响应封装，复杂逻辑放入 service。不同上游系统通过 provider client 接口适配。 前端按 API 类型、页面容器、业务组件、纯工具函数拆分，避免把所有交互堆在单个视图文件。 

Frontend
  ├─ ModelSquareView
  ├─ UpstreamGroupsView
  ├─ PaymentView
  ├─ HelpView
  └─ API clients

Backend
  ├─ SettingService / PaymentConfigService
  ├─ UpstreamManagementService
  ├─ UpstreamProviderClient implementations
  ├─ LLMMonitorService
  ├─ RechargeOptionService
  ├─ AccountRateGuardService
  └─ HTTP handlers / routes

配置来源采用“数据库 settings 优先，config.yaml 作为默认值”的规则。敏感字段只允许写入，不在读取接口中返回明文。

3. 上游管理 / 模型广场

3.1 功能说明

上游管理负责连接外部模型平台或上游 sub2api/new-api 实例，拉取模型、分组、key 信息，并把这些信息和本地分组关联。 模型广场是用户侧展示层，展示可用模型、价格、模式、所属分组、倍率等信息。

3.2 后端模块

模块 职责
UpstreamManagementService 业务入口，聚合 provider 配置、上游客户端、本地分组，输出模型广场和管理页所需数据。
UpstreamProviderClient 接口，定义 FetchModels、FetchGroups、FetchKeys、Test。
Sub2APIProviderClient 适配 sub2api 风格上游。
NewAPIProviderClient 适配 new-api 风格上游。
UpstreamConfigService 读取和规范化 provider 配置，处理密码保留、默认值、启停状态。
UpstreamManagementHandler 注册 HTTP API，调用 service 并返回标准响应。

3.3 数据模型

{
  "id": "primary",
  "name": "Primary Upstream",
  "type": "sub2api",
  "enabled": true,
  "base_url": "https://upstream.example.com",
  "login_url": "/api/user/login",
  "models_url": "/api/channel/models",
  "groups_url": "/api/groups",
  "keys_url": "/api/tokens",
  "email": "admin@example.com",
  "password_configured": true,
  "sort_order": 10
}

3.4 API 设计

接口 说明
GET /api/v1/admin/upstream-management/providers 读取 provider 配置，不返回密码明文。
POST /api/v1/admin/upstream-management/providers/test 用表单中的 provider 配置测试登录、模型、分组、key 接口。
GET /api/v1/admin/upstream-management/models 管理员查看上游模型原始聚合结果。
GET /api/v1/admin/upstream-management/groups 管理员查看上游分组和本地分组匹配结果。
GET /api/v1/admin/upstream-management/key-summary 按上游分组汇总 key 数量和 key 名称。
POST /api/v1/admin/upstream-management/sync-rates 把上游更高的分组倍率同步到本地分组。
GET /api/v1/model-square 用户侧模型广场数据，只返回用户需要的信息。

3.5 匹配规则

  • 分组匹配默认按名称规范化后匹配：去空格、转小写、折叠连续空白。
  • 上游分组未匹配本地分组时仍展示，但标记为 unmatched。
  • 同一个本地分组匹配多个上游分组时，倍率同步使用最高上游倍率。
  • 倍率同步只允许调高本地倍率，不自动调低。

3.6 前端页面

  • ModelSquareView：用户侧模型列表，支持搜索、按分组过滤、按 provider 过滤、复制模型 ID。
  • UpstreamGroupsView：管理员页面，支持搜索、平台过滤、匹配状态过滤、key 摘要弹窗、倍率同步、监控趋势摘要。
  • SettingsView：只保留 provider 配置表单和测试按钮。

4. LLM 监控

4.1 功能说明

LLM 监控用于接入外部状态 API，后端代理请求，按本地分组过滤 provider，移除敏感字段后返回给前端。 管理页只展示与上游分组相关的简要趋势，完整展示继续使用独立静态页。

4.2 后端模块

模块 职责
LLMMonitorService 代理、过滤、脱敏、错误映射。
LLMMonitorClient 访问外部 status API，处理超时和请求参数。
LLMMonitorFilter 根据本地分组生成允许 provider 集合，过滤 groups 和 meta IDs。

4.3 API 设计

接口 说明
GET /api/llm-monitor/status?period=90m&board=hot 公开或半公开监控代理，返回过滤后的上游状态数据。
GET /api/llm-monitor/groups 返回可展示的监控分组元信息。
GET /api/v1/admin/upstream-management/monitor-status 管理侧使用，标准 API 响应结构。

4.4 规则

  • 默认 period=90m，默认 board=hot。
  • 监控上游 URL 必须是 http/https 绝对 URL，不能包含 fragment。
  • 上游请求超时建议 15 秒。
  • 上游非 2xx 状态原样透传状态码和 body，但不写缓存。
  • 响应脱敏移除已知敏感域名、内部 provider 标识、未匹配本地分组的数据。

5. 充值金额可配置

5.1 功能说明

管理员可以配置充值档位、首充优惠、帮助图片和帮助文案。用户侧充值页根据后端返回的配置渲染，不自行计算到账金额。

5.2 配置字段

字段 说明
payment_recharge_options 普通充值档位数组，例如 [10, 20, 50, 100]。
payment_intro_recharge_pay_amount 首充优惠实付金额。
payment_intro_recharge_credit_amount 首充优惠到账金额。
payment_help_text 充值页帮助文案。
payment_help_image_url 充值页帮助图片 URL。

5.3 业务规则

  • 普通档位必须大于 0，保存时去重并升序。
  • 首充优惠只有 credit_amount > pay_amount > 0 时启用。
  • 首充优惠只对没有领取过首充的用户展示。
  • 已取消、已过期、失败订单不算领取过首充；支付中和已成功订单算领取过。
  • 创建订单时，后端根据配置计算实付金额和到账金额，不能信任前端传入的到账金额。

5.4 API 设计

  • GET /api/v1/payment/config：返回支付开关、帮助文案、帮助图片等。
  • GET /api/v1/payment/recharge-options：返回当前用户可见档位，包含首充优惠状态。
  • PUT /api/v1/admin/payment/config 或 PUT /api/v1/admin/settings：保存管理员配置。

6. 账号倍率守护

6.1 功能说明

账号倍率守护用于发现并处理成本倒挂风险：本地账号绑定了低倍率分组，但上游 key 所属分组倍率更高。 它支持模拟检查和正式运行。模拟检查只报告风险，正式运行会解除低倍率分组绑定并写入审计记录。

6.2 后端模块

模块 职责
AccountRateGuardService 执行倍率检查、风险判断、解绑、审计记录。
AccountRateGuardScheduler 按配置间隔定时运行正式检查。
AccountRateGuardRepository 持久化运行记录和解绑审计。

6.3 API 设计

接口 说明
POST /api/v1/admin/upstream-management/account-rate-guard/run 手动运行，body 支持 {"dry_run": true}。
GET /api/v1/admin/upstream-management/account-rate-guard/status 读取最近一次运行状态和结果。
GET /api/v1/admin/upstream-management/account-rate-guard/audits 读取最近解绑审计记录。

6.4 匹配与处理规则

  • 默认按本地账号名和上游 key 名规范化后匹配。
  • 规范化规则与分组名称一致：去空格、转小写、折叠连续空白。
  • 如果上游 key 倍率高于本地账号任一绑定分组倍率，则生成风险项。
  • dry-run 不写任何账号变更，也不写解绑审计。
  • 正式运行只解除低于上游倍率的分组绑定，不影响满足倍率要求的分组。
  • 正式解绑写审计：run_id、account_id、account_name、provider、upstream_key、local_rate、upstream_rate、unbound_group_ids。

7. 帮助文档

第一版采用静态 HTML 方案，降低实现复杂度。前端新增用户侧 /help 路由，页面内 iframe 加载同源 /help.html。 构建产物和部署目录都需要包含该文件。

  • frontend/public/help.html：前端静态资源。
  • frontend/src/views/user/HelpView.vue：iframe 容器。
  • deploy/help.html：部署示例使用。
  • CSP 需要允许同源 iframe，并确保帮助页不会被默认安全头阻断。
后台编辑帮助文档属于后续独立功能，不进入第一版，避免引入富文本、版本管理和权限审计复杂度。

8. 设置项与兼容策略

新开发使用统一命名，旧命名仅在迁移期读取兼容，不继续扩散到新代码。

前缀 用途
upstream_management_* 上游 provider、模型、分组、key、登录相关配置。
llm_monitor_* 监控 status API、标题、provider URL。
payment_recharge_* 充值档位、首充优惠、到账金额规则。
account_rate_guard_* 倍率守护开关、运行间隔、审计保留数量。

密码类字段保存时如果为空表示保留旧值；读取时只返回 password_configured。

9. 错误处理

  • 上游 provider 测试接口返回分阶段结果：登录、模型、分组、key，每阶段包含成功状态、状态码、错误摘要。
  • 模型广场用户接口在单个 provider 失败时可以降级返回其他 provider 数据，并在管理员接口中暴露失败详情。
  • LLM 监控上游失败返回 502；配置 URL 非法返回 500 或管理侧校验错误。
  • 充值配置保存时返回明确字段错误，例如档位为空、首充到账小于等于实付。
  • 账号倍率守护单 provider 失败不阻断其他 provider，最终结果中记录 provider error。

10. 测试计划

10.1 后端测试

  • provider client：登录成功/失败、模型拉取、分组拉取、key 拉取、非 2xx 响应。
  • 上游分组匹配：大小写、空格、未匹配、多上游分组匹配同一本地分组。
  • 倍率同步：只调高、不调低、空分组、不存在本地分组。
  • LLM 监控：URL 校验、provider 过滤、敏感字段脱敏、上游超时、上游 500。
  • 充值配置：档位去重排序、首充可用性、订单金额计算、重复领取拦截。
  • 倍率守护：dry-run 不写入、正式运行解绑、审计记录、provider 部分失败。

10.2 前端测试

  • API wrapper：请求路径和参数正确。
  • ModelSquareView：搜索、provider 过滤、分组过滤、复制 model ID。
  • UpstreamGroupsView：匹配状态、key 摘要、倍率风险、守护运行状态。
  • PaymentView：普通档位、首充档位、帮助文案和图片。
  • HelpView：iframe 指向 /help.html。

11. 开发顺序

  1. 定义 settings 字段、DTO、配置读取和默认值。
  2. 实现 provider client 接口和 sub2api/new-api 适配。
  3. 实现上游管理 service、模型广场用户 API、管理侧分组 API。
  4. 实现模型广场页面和上游分组管理页面。
  5. 实现 LLM 监控代理、过滤、脱敏和管理页趋势摘要。
  6. 实现充值金额配置、首充优惠和充值页展示。
  7. 实现账号倍率守护 service、API、定时任务和审计展示。
  8. 接入帮助文档、侧边栏导航、CSP 和部署配置。
  9. 执行后端单元测试、前端单元测试、构建验证。

12. 评审检查清单

  • handler 是否只做 HTTP 层职责。
  • provider client 是否可以独立单元测试。
  • 敏感配置是否没有明文返回。
  • 用户侧模型广场是否不会暴露管理字段。
  • LLM 监控是否做了 URL 校验和脱敏。
  • 充值订单金额是否以后端计算结果为准。
  • 账号倍率守护正式运行是否有审计记录。
  • 帮助页是否在生产构建和部署目录中都可访问。
该设计文档聚焦可重新开发的功能边界，不要求复用当前分支里的具体实现文件。后续实施计划应按本设计拆成可独立验证的小任务。
